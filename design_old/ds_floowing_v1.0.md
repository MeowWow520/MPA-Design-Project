## 掌上游戏机（俄罗斯方块）—— 框架设计 v1.0

---

### 一、你的设计方案合理性分析

#### 1.1 正确的部分

| 项目 | 评价 |
|------|------|
| 游戏类型选俄罗斯方块 | 复杂度适中，适合 1 周课设 |
| 主循环拆成 HandleEvents / Update / Render / Detect | 模块化思路正确，符合游戏循环范式 |
| 两片 8×8 矩阵做显示 | 16×8 或 8×16 的像素空间，可承载简版俄罗斯方块 |
| 用 80C51/80C52 | Proteus 仿真支持良好，资料丰富 |

#### 1.2 需要修正/补充的问题

| 编号 | 问题 | 说明 |
|------|------|------|
| ① | **main() 没有死循环** | 8051 无操作系统，main 执行完就"跑飞"或复位。必须 `while(1){}` 包裹主循环。 |
| ② | **没有初始化步骤** | 缺系统初始化：端口方向、定时器、中断、游戏初始状态、随机种子等。 |
| ③ | **缺少重力/自动下落机制** | 俄罗斯方块的核心是方块定时自动下移。你的循环里完全没有定时概念。需要用定时器中断来驱动"下落 tick"。 |
| ④ | **缺少新方块生成逻辑** | Detect() 消除行后 / 当前方块触底后，应产生新方块 (Spawn)。 |
| ⑤ | **缺少碰撞检测** | 左右移动、旋转、下落都必须检测是否与已有方块或边界碰撞。 |
| ⑥ | **缺少游戏结束判定** | 新方块出生位置被阻挡 → Game Over。 |
| ⑦ | **Detect() 与 Render() 顺序问题** | Detect 是逻辑层（消除行），放在 Render 前更合理，但当前顺序也可工作——只是渲染的是"消行前"的画面。建议放在 Update 之后、Render 之前。 |
| ⑧ | **没有显示刷新机制** | LED 点阵必须动态扫描（逐行刷新）。扫描频率 >50Hz。需要用定时器中断实现，不能放在主循环里串行执行。 |
| ⑨ | **按键无消抖** | 机械按键需要软件消抖（延时或状态机），否则一次按下可能触发多次。 |
| ⑩ | **16×8 vs 8×16 方向选择** | 两块 8×8 矩阵如何拼接决定了游戏场地形状，直接影响 Tetris 体验。下文详细讨论。 |

#### 1.3 修正后的主循环结构

```
main():
    系统初始化 → 开全局中断 → while(1):
        HandleEvents()   ← 读取按键，更新动作标志
        Update()         ← 根据标志移动/旋转/下落
        Detect()         ← 消行判断与执行
        Render()         ← 更新显示缓冲区
```

其中 **下落 tick** 和 **显示扫描** 均由定时器中断驱动，不阻塞主循环。

### 二、参考方案与建议

#### 2.1 矩阵摆放方向的选择（关键决策）

两块 8×8 矩阵有两种拼接方式：

| 方式 | 分辨率 | 游戏场尺寸 | Tetris 适配度 |
|------|--------|-----------|--------------|
| A：左右并排 | 16 列 × 8 行 | 太"扁"，方块刚出生就到底 | **不适合** |
| B：上下堆叠 | 8 列 × 16 行 | 窄但高，传统 Tetris 是 10×20，缩到 8×16 可行 | **推荐** |

**结论**：采用 **B（上下堆叠，8×16）**。所有标准方块最大宽度 4，8 列足够；16 行高度可提供一定游玩空间。

#### 2.2 方案 A：你当前的架构 + 完善（逻辑驱动式）

你的主循环思路 + 定时器中断配合，是典型的"游戏状态机"架构。
- 优点：逻辑清晰，易于理解和调试
- 适用：对实时性要求不极端的场景

核心数据结构：
```
uint8_t board[16];        // 游戏板，每行 1 字节，bit7..bit0 对应 8 列
                          // board[0] = 顶行, board[15] = 底行
struct {
    uint8_t x, y;         // 方块左上角坐标 (列, 行)
    uint8_t type;         // 0-6 表示 I/O/T/S/Z/J/L
    uint8_t rotation;     // 0-3
} piece;

uint8_t disp_buf[16];    // 显示缓冲区 = board + 当前 piece 叠加
```

定时器分配：
- **T0**：显示扫描定时器（~1ms 中断，16 行 × 1ms = 16ms/帧 ≈ 62Hz）
- **T1**：重力下落定时器（~500ms 中断，可随等级加快）

#### 2.3 方案 B：全中断驱动式（备选）

所有逻辑在定时器中断中完成，主循环只做 `while(1);` 或休眠。
- 优点：时序精确
- 缺点：中断服务程序过长可能影响扫描稳定性
- 不推荐给课设级别项目，调试困难

#### 2.4 对比总结

| 维度 | 方案 A（推荐） | 方案 B |
|------|--------------|--------|
| 复杂度 | 中 | 高 |
| 可维护性 | 好 | 差 |
| 课设适用 | 是 | 否 |
| 显示扫描稳定性 | 好（中断只做扫描） | 差（中断做太多事） |

### 三、IO 口分配方案

#### 3.1 硬件顶层框图

```
80C51/80C52
    │
    ├── P1 ───→ 74HC595 ×1 ──→ 8 列阳极（经限流电阻）
    │
    ├── P2.0~P2.2 ───→ 74HC595 ×2 (级联) ──→ ULN2803 ×2 ──→ 16 行阴极
    │
    ├── P3.0~P3.4 ──← 按键（左/右/下/旋转/开始）
    │
    └── P3.5 ───→ 无源蜂鸣器（可选）
```

#### 3.2 你的方案 —— IO 口分配（直接端口驱动版）

若不用 74HC595，用 MCU 端口直驱：

| 端口 | 位 | 功能 | 方向 | 备注 |
|------|-----|------|------|------|
| **P0** | 0-7 | 上矩阵 8 行阴极 | 输出 | 需外部上拉电阻，经 ULN2803 驱动 |
| **P1** | 0-7 | 8 列阳极 | 输出 | 每线串 220Ω 限流电阻 |
| **P2** | 0-7 | 下矩阵 8 行阴极 | 输出 | 经 ULN2803 驱动 |
| **P3** | 0 | LEFT 按键 | 输入 | 外部上拉，按下为 0 |
| | 1 | RIGHT 按键 | 输入 | 同上 |
| | 2 | DOWN 按键 | 输入 | 同上，可复用 INT0 |
| | 3 | ROTATE 按键 | 输入 | 同上，可复用 INT1 |
| | 4 | START 按键 | 输入 | 同上 |
| | 5 | 蜂鸣器 | 输出 | 可选 |
| | 6-7 | 保留 | — | WR/RD，不用外部存储时可作 IO |

**该方案缺点**：
- P0 需外部上拉，Proteus 中需接排阻
- 占用 3 个端口用于显示，仅剩 P3 给按键
- MCU 直驱 LED 阳极电流能力有限（~10mA），亮度可能偏低
- 端口使用率极高，扩展性差

#### 3.3 我的推荐方案 —— IO 口分配（74HC595 驱动版）

| 端口 | 位 | 功能 | 方向 | 备注 |
|------|-----|------|------|------|
| **P1** | 0 | COL_SER (列数据) | 输出 | → 74HC595(1) DS |
| | 1 | COL_SRCLK (移位时钟) | 输出 | → 74HC595(1) SH_CP |
| | 2 | COL_RCLK (锁存时钟) | 输出 | → 74HC595(1) ST_CP |
| | 3 | ROW_SER (行数据) | 输出 | → 74HC595(2) DS |
| | 4 | ROW_SRCLK | 输出 | → 74HC595(1)(2) SH_CP 并联 |
| | 5 | ROW_RCLK | 输出 | → 74HC595(1)(2) ST_CP 并联 |
| | 6-7 | 保留 | — | — |
| **P3** | 0 | LEFT | 输入 | 外接上拉 |
| | 1 | RIGHT | 输入 | 外接上拉 |
| | 2 | DOWN | 输入 | 外接上拉，复用 INT0 |
| | 3 | ROTATE | 输入 | 外接上拉，复用 INT1 |
| | 4 | START/PAUSE | 输入 | 外接上拉 |
| | 5 | BUZZER | 输出 | 可选 |
| | 6-7 | 保留 | — | — |
| **P0** | 0-7 | 保留 | — | — |
| **P2** | 0-7 | 保留 | — | — |

**优点**：
- 显示仅用 6 根线
- 保留 P0、P2 整个端口及 P1 部分引脚用于扩展
- 74HC595 输出驱动能力足够，经 ULN2803 行扫电流充裕
- 显示扫描在中断中完成，不占用主循环时间

**外围器件清单**：
- 74HC595 × 3（1 片管 8 列阳极，2 片级联管 16 行阴极）
- ULN2803 × 2（每片 8 路达林顿，驱动阴极）
- 8×8 点阵 × 2（建议绿色，共阳型）
- 电阻：8×220Ω（列限流），4×10kΩ（按键上拉）
- 按键 × 5
- 无源蜂鸣器 + 三极管驱动（可选）
- 12MHz 晶振 + 2×30pF 电容
- 10μF + 10kΩ 复位电路

### 四、计数器/定时器分配

| 资源 | 用途 | 工作模式 | 初值 / 周期 |
|------|------|---------|------------|
| **T0** | 显示扫描 | 模式 1 (16 位定时器) | ~1ms 中断周期 |
| **T1** | 重力下落 | 模式 1 (16 位定时器) | ~500ms 中断周期，可变 |
| **外部中断 INT0** | 下落加速（可选） | 下降沿触发 | 按下 DOWN 键直接触发一次下落 |
| **外部中断 INT1** | 旋转（可选） | 下降沿触发 | 预留 |

**T0 显示扫描参数**：
- 晶振 12MHz → 机器周期 1μs
- 16 行 × 1ms = 16ms/帧 ≈ 62.5Hz（不闪烁）
- 初值：65536 - 1000 = 64536 → TH0=0xFC, TL0=0x18

**T1 重力参数**：
- 500ms = 500000 机器周期
- 65536 - 50000 = 15536 × 10 次溢出 = 500ms
- 初值：TH1=0x3C, TL1=0xB0（50ms × 10 次 = 500ms）

### 六、其他建议方案

#### 6.1 显示方案替代：MAX7219

如果 Proteus 元件库中有 MAX7219（或可用兼容模型）：
- 仅需 3 线 (DIN, CLK, LOAD)
- 内置 BCD 译码、扫描、恒流驱动
- 可驱动 8 位 8 段数码管或 8×8 点阵
- 两块矩阵用两片 MAX7219 级联
- **优点**：硬件极简，软件扫屏无需中断
- **缺点**：需 SPI 时序编程，成本较高

#### 6.2 显示方案替代：74HC138 + 74HC595

用 74HC138（3-8 译码器）代替行扫描 74HC595：
- 2 片 74HC138 → 16 行选择（4 根地址线）
- 1 片 74HC595 → 8 列数据（3 根线）
- 总计 7 根线
- 比全 595 省 2 根线，但 138 只有一路输出为低

#### 6.3 按键方案替代：ADC 键盘

将所有按键接在同一 ADC 通道上，不同按键接不同分压电阻：
- 仅需 1 个 ADC 引脚（但标准 80C51 无 ADC，需外挂或选 STC 型号）
- 对 80C51/80C52 不适用（无内置 ADC）

#### 6.4 面向"加分项"的建议

- 显示下一个方块（Next Piece）—— 可在 16 行中留出顶部 4 行做预览区
- 分数显示 —— 若有剩余端口，可加 4 位数码管（用 74HC595 + TM1637 之类）
- 等级加速 —— 分数越高，T1 定时间隔越短
- 音效 —— 蜂鸣器在消行、移动、旋转时发出不同频率/时长的鸣叫
